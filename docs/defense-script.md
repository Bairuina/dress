# Dress 项目答辩稿

## 1. 开场介绍

大家好，我这次做的项目是一个面向个人用户的电子衣橱与穿搭管理系统。

这个项目不只是一个简单的衣物记录工具，我希望它能覆盖用户从管理已有衣物，到组合穿搭，再到分享内容和获取购买灵感的完整链路。

所以我把它设计成了一个兼顾：

- 衣橱管理
- 穿搭管理
- 穿搭分享
- 灵感穿搭
- 商业化扩展

的全栈项目。

## 2. 我为什么做这个项目

我选择这个方向主要有两个原因。

第一，这个产品场景比较真实，用户痛点比较明确：

- 不知道自己有哪些衣服
- 有很多衣服但不会搭
- 想让朋友帮忙参考穿搭
- 想先看搭配效果，再决定买什么

第二，这个项目很适合作为全栈工程考核项目，因为它既有完整业务闭环，也有比较清晰的前后端分层和后续扩展空间。

## 3. 产品定位

这个项目当前的定位是：

一个以个人衣橱为基础，连接穿搭管理、内容分享、购买转化和后续 AI 辅助能力的穿搭平台。

我把产品拆成了几个核心模块：

- 用户系统
- 我的衣橱
- 我的穿搭
- 穿搭分享
- 灵感穿搭

其中穿搭分享除了公开发布外，我还考虑了私域传播场景，比如一键分享给闺蜜，用来征求搭配意见。

## 4. 核心业务闭环

从业务角度，这个项目的主线是：

1. 用户登录注册
2. 录入自己的衣物
3. 对衣物做分类、颜色、季节管理
4. 基于已有衣物组合穿搭
5. 发布穿搭分享内容
6. 后续扩展灵感穿搭和商品推荐位

这样项目不只是若干独立页面，而是一个完整的产品闭环。

## 5. 技术选型

前端我采用的是：

- React
- TypeScript
- Vite
- Ant Design

后端采用的是：

- FastAPI
- SQLAlchemy
- MySQL

缓存层预留并接入了：

- Redis

后续工程化方向预留了：

- Docker
- Nginx

我这样选择的原因是：

- React + TS 是我比较熟悉的前端栈，开发效率更高
- FastAPI 适合快速开发接口，同时结构清晰，便于我学习后端分层
- MySQL 适合作为核心业务数据存储
- Redis 适合作为后续缓存和会话层

## 6. 系统架构说明

这个项目是标准的前后端分离架构。

前端主要负责：

- 页面展示
- 表单交互
- 请求后端接口
- 状态管理

后端主要负责：

- 接口定义
- 参数校验
- 业务逻辑
- 数据库访问
- 缓存处理

后端我做了比较清晰的分层：

- `api/routes`
  负责接口路由
- `schemas`
  负责请求和响应的数据结构
- `services`
  负责业务逻辑编排
- `crud`
  负责数据库读写
- `models`
  负责数据库表结构定义
- `db`
  负责数据库和 Redis 连接

我这样拆分的目的是让代码职责更清晰，也更利于测试和后续维护。

## 7. 数据库设计思路

数据库我目前围绕几张核心表展开：

- `users`
- `clothes`
- `outfits`
- `outfit_items`
- `posts`
- `comments`
- `post_likes`
- `post_favorites`
- `inspiration_outfits`
- `inspiration_items`

其中有几个关键关系：

- 一个用户可以有多件衣物
- 一个用户可以有多套穿搭
- 一套穿搭和多件衣物之间是多对多关系，所以我用了 `outfit_items` 作为中间表
- 一条穿搭分享可以关联一套穿搭

这个设计既能满足当前 MVP，也方便后面继续扩展分享、推荐和商业化能力。

## 8. Redis 在项目里的作用

Redis 我没有把它只理解成“缓存登录”，而是把它放在一个更完整的后端架构里理解。

在这个项目里：

- MySQL 负责正式业务数据
- Redis 负责热点缓存和临时状态

当前项目里，Redis 已经预留并接入了缓存层，适合后续放：

- 衣物列表缓存
- 首页统计缓存
- 验证码
- 会话状态
- token 黑名单
- 点赞数和浏览数等热点计数

这样我的理解会更接近真实项目，而不是只把 Redis 当成“登录缓存”。

## 9. 工程思考

在实现这个项目时，我不只是关注页面能不能显示、接口能不能返回，而是尽量从工程角度去搭结构。

我重点考虑了这些问题：

- 前后端边界是否清晰
- 数据结构是否可维护
- 路由、业务逻辑、数据库访问是否分层
- 环境配置是否隔离
- 后续是否方便接入测试
- 是否有 Redis、Docker、Nginx 的扩展空间

我认为作为一个全栈项目，工程规范和可解释性和功能本身同样重要。

- 前后端本地环境打通

这意味着这个项目已经不是一个单纯的静态展示，而是一个具备实际开发基础的全栈项目骨架。

## 12. 总结

一句话总结这个项目：

这是一个以前后端分离为基础的电子衣橱与穿搭管理系统，它不仅覆盖了核心业务闭环，也考虑了内容分享、商业化路径和后续工程扩展能力。

如果继续推进，我希望它不只是一个考核项目，而是一个真正可以继续产品化演进的基础版本。
